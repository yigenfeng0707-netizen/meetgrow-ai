## 二、需求池总览

### 2.1 需求来源分析

V1.1的需求来源分为四类：

**用户反馈驱动**（正式开售后收集）：预计高频需求集中在"生成后想改内容"、"某个模块想重来"、"有没有更多模板"三个方向。

**竞品对标驱动**：对标ChatGPT/通义千问/文心一言等AI写作工具，它们在"生成→编辑→再生成"的交互循环上做得更成熟。

**商业化驱动**：提升产品壁垒和付费转化率，增加续费理由。

**技术债驱动**：激活码安全加固、存储扩容、代码模块化。

## 三、P0 核心需求（必做，预计4-5天）

### F01 模块内容编辑

**用户故事**：作为用户，生成方案后我想直接在页面上修改某个模块的内容，而不是必须导出到Word再改。

**现状**：生成完成后，模块内容区是纯展示的`<div class="module-content">`，只能查看和复制。

**设计方案**：

1. 每个模块Tab增加"编辑"按钮，点击后内容区变为`contenteditable`或`<textarea>`
2. 编辑状态显示"保存"和"取消"按钮
3. 保存时更新`state.currentPlan.content[id]`并同步到localStorage
4. 编辑后的内容参与Word/Markdown/PDF导出
5. 标记已编辑的模块（Tab上显示小标记"已修改"）

**技术要点**：
- 使用`contenteditable="true"`方案比`<textarea>`更好，因为能保留HTML格式
- 需要在保存时将innerHTML转回markdown存储（或同时保存两种格式）
- 简单方案：编辑时显示markdown源码在textarea中，保存时重新渲染

**预估工时**：1天

---

### F02 单模块重新生成

**用户故事**：作为用户，当某个模块的输出不满意时，我想只重新生成这一个模块，而不是整个方案重来。

**现状**：只有"开始生成增长方案"按钮触发全流程，没有单模块重新生成的入口。但已有`requestContinuation`函数可作为参考（它实现了"补写缺失模块"的逻辑）。

**设计方案**：

1. 每个模块Tab的标题栏增加"重新生成"按钮（🔄 图标）
2. 点击后弹出确认框："重新生成将覆盖当前内容，是否继续？"
3. 确认后调用AI接口，仅请求生成该模块
4. Prompt设计：将原方案的其他模块作为上下文提供，要求只输出当前模块
5. 流式渲染替换当前模块内容
6. 更新进度条（单模块模式）

**Prompt模板**：
```
你之前为活动「{活动名}」生成了招商增长方案。以下模块已完成：
{其他模块的概要}

现在请重新生成模块「{模块编号}. {模块名}」，要求：
1. 与其他模块保持逻辑一致性
2. 注意以下用户反馈：{可选的用户反馈输入框}
3. 内容至少500字，包含分析+话术模板+执行清单
4. 以 ## {模块编号}. {模块名} 开头
```

**技术要点**：
- 复用现有SSE流式处理逻辑
- 需要修改`parseAndRenderModules`支持只更新单个模块
- 添加`isGenerating`状态的单模块变体

**预估工时**：1.5天

---

### F03 扩展预设模板（4→10套）

**用户故事**：作为用户，我的活动场景不止峰会/园区/培训/沙龙，还可能是年会、发布会、招商路演等，希望能一键填充对应的表单。

**现状**：TEMPLATES对象中只有4套预设（summit/park/training/salon），都是硬编码在HTML中。

**新增模板**：

| 编号 | 模板名称 | 适用场景 | 关键参数 |
|------|----------|----------|---------|
| T05 | 年度行业盛典 | 年会/颁奖典礼/行业盛典 | 800-2000人，赞助+门票+品牌曝光 |
| T06 | 新品发布会 | 产品发布/品牌升级/战略发布 | 100-500人，媒体传播+渠道招商 |
| T07 | 招商路演 | 城市巡回招商/项目路演 | 50-100人/站，项目融资+渠道合作 |
| T08 | 行业展览/博览会 | 展会/博览会/交易会 | 500-5000人，展位销售+观众引流 |
| T09 | 私域增长活动 | 线上直播/社群活动/裂变活动 | 不限人数，私域沉淀+社群裂变+转化 |
| T10 | 企业内训/咨询 | 企业定制培训/顾问项目 | 20-50人，内训签约+年度顾问 |

**设计方案**：
1. 在TEMPLATES对象中新增6套模板数据
2. 表单区域的模板选择改为两列网格（目前单列4个按钮）
3. 每个模板卡片显示场景图标+适用标签
4. SYSTEM_PROMPT中增加"根据活动类型自适应调整输出风格"的指令

**技术要点**：
- 纯数据层变更，无架构改动
- 模板选择UI需要适配更多选项（grid布局调整）
- 考虑将TEMPLATES提取到单独的JSON配置（为F08做准备）

**预估工时**：0.5天

### F04 表单配置保存（方案草稿）

**用户故事**：作为用户，我填好了一个复杂的活动表单，想保存为"草稿"，下次可以直接加载使用，不用重复填写。

**现状**：表单数据不持久化，刷新页面后丢失。TEMPLATES是硬编码的，用户无法添加自定义模板。

**设计方案**：

1. 输入表单底部增加两个按钮："保存草稿"和"加载草稿"
2. 草稿存储到localStorage（key: `mg_drafts`），与历史方案分开管理
3. 草稿列表弹窗：显示已保存的草稿（名称+日期+类型），支持加载和删除
4. 草稿数据结构：`{id, name, type, audience, scale, goals, extra, modules[], createdAt}`
5. 最多保存10个草稿

**技术要点**：
- 复用localStorage存储机制
- 草稿与历史方案使用不同的namespace避免冲突
- 加载草稿时填充表单字段+选中对应模块

**预估工时**：0.5天

## 四、P1 重要需求（建议做，预计3-4天）

### F05 深色模式

**用户故事**：作为用户，长时间使用产品时希望有深色模式减少眼睛疲劳，并且深色模式是现代产品的标配。

**设计方案**：

1. 设置弹窗中增加"主题切换"选项（浅色/深色/跟随系统）
2. 定义深色主题的CSS变量集合：
   ```css
   [data-theme="dark"] {
     --bg: #0f172a;
     --surface: #1e293b;
     --surface-alt: #334155;
     --text: #f1f5f9;
     --text-secondary: #94a3b8;
     --border: #334155;
     /* ... */
   }
   ```
3. 通过`<html data-theme="dark">`切换主题
4. 存储用户偏好到localStorage（key: `mg_theme`）
5. 确保所有组件（模态框、侧边栏、输出区、表格、代码块）在深色模式下可读

**技术要点**：
- CSS变量方案已具备，只需增加dark变量集
- 需要逐一检查所有`hardcoded color`（如`style="color:#..."`）并改为变量
- agreement.html也需要同步适配

**预估工时**：1.5天

### F06 AI润色助手

**用户故事**：作为用户，AI生成的某个模块大体方向对了，但某些段落我想让它"写得更好"——更专业、更有说服力、更简洁——而不是完全重写。

**现状**：没有对已生成内容进行AI辅助优化的能力。

**设计方案**：

1. 模块内容区（编辑模式下）增加"AI润色"按钮
2. 用户可以先手动选中一段文字，然后点击"AI润色"
3. 弹出润色选项面板：
   - 润色风格：更专业 / 更有说服力 / 更简洁 / 更生动
   - 润色范围：选中文字 / 整个模块
4. 调用AI接口进行润色，流式展示结果
5. 用户确认后替换原内容

**Prompt模板**：
```
请对以下招商方案内容进行润色优化。

润色要求：{风格选择}
润色范围：{选中文字或整个模块}

原始内容：
{content}

请保持原意不变，只优化表达。直接输出润色后的内容，不需要解释。
```

**技术要点**：
- 新增一个轻量级AI调用函数（非完整方案生成）
- 支持文字选区获取（`window.getSelection()`）
- 润色结果需要二次确认后才替换（避免误操作）

**预估工时**：1.5天

---

### F07 PPT导出

**用户故事**：作为用户，我需要把招商增长方案直接用于向领导汇报或向客户路演，PPT格式最方便。

**现状**：导出格式为Word/Markdown/PDF(打印)/复制全文，没有PPT。

**设计方案**：

1. 导出按钮区新增"📊 PPT"按钮
2. 使用PptxGenJS库（CDN引入，约100KB）在浏览器端生成.pptx文件
3. PPT结构设计：
   - 封面页：活动名称 + 类型 + 日期 + MeetGrow AI Logo
   - 目录页：列出所有模块
   - 每个模块1-2页：标题+要点+关键数据
   - 结尾页：联系方式 + CTA
4. 自动将markdown内容精简为PPT要点（提取h2/h3/加粗文字/列表）

**技术要点**：
- 引入PptxGenJS：`<script src="https://cdn.jsdelivr.net/gh/gitbrent/PptxGenJS/dist/pptxgen.bundle.js">`
- markdown→PPT内容转换需要自定义逻辑
- 单文件HTML中引入外部库需要注意离线可用性
- PPT样式需要与产品品牌色一致（#0ea5e9主色 + #22c55e辅色）

**预估工时**：1天

## 五、P2 改进需求（按需选择，预计2-3天）

### F08 历史存储扩容 + 方案标签

**现状**：localStorage存20条历史，单条方案约20-30KB，20条就接近500KB的localStorage软上限。

**改进方案**：
1. 历史存储改为分页：最近20条在内存中渲染，更早的按需加载
2. 方案增加标签系统：用户可以为方案打标签（如"客户A"、"终版"、"待修改"）
3. 历史列表支持按标签筛选和关键词搜索
4. 当localStorage接近满时自动提示并支持导出全部历史为JSON备份文件

**预估工时**：1天

---

### F09 断点续传增强（多轮续写）

**现状**：`requestContinuation`只执行1轮续写。如果AI输出再次中断，不会继续。

**改进方案**：
1. 续写后再次检测缺失模块，如有则触发第2轮续写
2. 最多自动续写3轮，超过3轮提示用户手动触发
3. 续写过程中显示"正在续写第X轮..."
4. 如果某模块连续3轮都无法生成完整内容，标记为"生成困难"并给出建议

**预估工时**：0.5天

---

### F10 方案版本回退

**用户故事**：作为用户，我编辑了某个模块后觉得改得不好，想回退到AI原始生成的版本。

**设计方案**：
1. 模块内容区增加"版本历史"入口（下拉菜单或弹窗）
2. 每次编辑保存时创建版本快照：`{version: 2, content: '...', timestamp: '...', type: 'user_edit'}`
3. 版本列表显示：V1(AI生成) → V2(用户编辑) → V3(AI润色) → ...
4. 点击任意版本可预览并恢复
5. 最多保留10个版本

**技术要点**：
- 在`state.currentPlan.content`基础上增加`state.currentPlan.versions`结构
- 需要控制版本数据大小，避免localStorage膨胀

**预估工时**：1天

## 六、P3 远期需求（V1.2+考虑）

以下需求技术复杂度高或依赖外部条件，建议在V1.2或更远期版本中实现：

### F11 在线激活码验证

**现状**：激活码通过客户端hash校验，理论上可以被逆向破解。

**远期方案**：
- 搭建轻量级验证服务（Cloudflare Workers / Vercel Edge Functions）
- 激活时POST到服务端验证，返回签名token
- 支持远程吊销（如发现盗版分发）
- 每次启动时静默验证token有效性
- 成本：Cloudflare Workers免费额度足够早期使用

**预估工时**：2天

---

### F12 多语言支持

**场景**：国际展会（如进博会、广交会）的招商方案需要中英双语。

**远期方案**：
- 表单增加"输出语言"选项（中文/英文/中英双语）
- SYSTEM_PROMPT根据语言选项切换
- UI国际化（i18n）作为更远期目标

**预估工时**：3天（含UI国际化）

---

### F13 批量导出

**场景**：用户生成了多套方案，想一次性全部导出给团队评审。

**远期方案**：
- 历史管理页面增加"批量选择→批量导出"功能
- 打包为ZIP文件下载
- 支持选择导出格式（全部Word或全部Markdown）

**预估工时**：1天

## 八、技术架构改进建议

### 8.1 代码模块化（为V1.1+做准备）

**现状**：所有代码（CSS + HTML + JS）在一个HTML文件中，约1600行。随着功能增加会变得难以维护。

**建议**：V1.1暂时维持单文件架构（降低部署复杂度），但做以下内部重构：
- 将MODULES、TEMPLATES、SYSTEM_PROMPT提取到文件顶部的独立配置区域
- JS函数按功能分组（UI管理 / API调用 / 数据处理 / 存储管理）
- 添加函数注释，标注每个函数的用途和依赖

**远期（V1.2+）**：考虑拆分为多文件（HTML + CSS + JS），通过构建工具打包。

### 8.2 存储架构升级

**现状**：全部依赖localStorage，容量约5-10MB。

**V1.1方案**：
- 历史方案压缩存储（移除rawOutput，只保留渲染后的content）
- 增加存储用量监控：`JSON.stringify(state.history).length` 超过4MB时预警
- 提供"导出备份/导入备份"功能（JSON文件）

**远期（V1.2+）**：
- IndexedDB替代localStorage（容量250MB+）
- 或对接云存储（需要后端服务）

### 8.3 AI调用层抽象

async function callAI(messages, onChunk, options = {}) {
  // 统一的SSE流式调用逻辑
  // 支持温度、max_tokens等参数配置
  // 统一的错误处理和重试机制

## 九、数据度量计划

### 9.1 V1.1需要追踪的核心指标

| 指标 | 定义 | 采集方式 | 目标 |
|------|------|---------|------|
| 模块编辑率 | 编辑过的模块占总生成模块的比例 | 前端埋点 | <30%（说明AI生成质量足够好） |
| 单模块重生率 | 重新生成过的模块比例 | 前端埋点 | <20%（说明首次生成满意度高） |
| 模板使用分布 | 各预设模板的使用占比 | localStorage统计 | 新模板合计占比>20% |
| 草稿保存率 | 保存草稿的用户比例 | localStorage统计 | >30%（说明功能有价值） |
| 润色使用频次 | 每个方案使用AI润色的平均次数 | 前端埋点 | 1-3次/方案 |
| PPT导出占比 | 使用PPT导出的用户比例 | 前端埋点 | >15% |
| 深色模式采用率 | 开启深色模式的用户比例 | localStorage | 参考行业均值30-40% |
| 方案留存率 | 7天后仍在使用产品的用户比例 | 统计平台 | >40% |

### 9.2 埋点方案

在`callAI`和各功能入口添加轻量埋点：
```javascript
// 功能使用埋点
function trackEvent(event, data = {}) {
  const log = JSON.parse(localStorage.getItem('mg_events') || '[]');
  log.push({ event, data, ts: Date.now() });
  if(log.length > 500) log.splice(0, log.length - 500);
  localStorage.setItem('mg_events', JSON.stringify(log));
  // 同时推送到百度统计
  if(window._hmt) _hmt.push(['_trackEvent', 'product', event, JSON.stringify(data)]);
}
```
